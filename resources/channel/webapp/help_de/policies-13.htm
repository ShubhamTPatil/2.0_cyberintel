Wenn Sie eine Policy mit Hilfe von Policy Manager erstellen oder bearbeiten, können Sie auch verschiedene Zeitpläne angeben. Die Zeitpläne definieren, wann Packages auf Ziele verteilt, aktualisiert, überprüft und repariert werden.

Hinweis: Das Planen von Package-Aktualisierungen kann Einfluss auf die Software auf den Clients haben und erfordert eine Berücksichtigung der Best Practices. Weitere Informationen dazu finden Sie unter Verwalten von Software, Daten und Aktualisierungen.

Dieser Abschnitt umfasst folgende Themen:

    • Welche Zeitpläne können für ein Package festgelegt werden?
    • Was sind Primär- und Sekundärzeitpläne?
    • Was sind Aktivierung und Verfall?
    • Was ist ein Aktualisierungszeitplan?
    • Was ist ein Reparaturzeitplan?
    • Was ist eine Wiederholung?

Welche Zeitpläne können für ein Package festgelegt werden?

Sie können für Packages in einer Policy folgende Zeitpläne festlegen:

    • Primärzeitplan
    • Sekundärzeitplan
    • Aktualisierungszeitplan
    • Reparaturzeitplan

Die vier Zeitplantypen sind voneinander unabhängig. Sie können für jeden Zeitplan Aktivierungs- und Verfallzeit sowie einen Zeitraum zu bestimmen, innerhalb dessen Packages für bestimmte Aktivitäten oder Zustände verfügbar sein sollen. Zeitpläne beziehen sich auf die Zeitzone des Clients.

Wenn Sie keine Aktivierungszeiten angeben möchten, können Sie auch festlegen, dass Aktivitäten bei der nächsten Policy Service-Aktualisierung stattfinden sollen. Wenn Sie diese Optionen auswählen, finden die Aktivitäten unmittelbar bei der nächsten Policy Service-Aktualisierung statt. Weitere Informationen zu Policy Service-Aktualisierungen finden Sie unter Angeben des Policy Service-Zeitplans für ein Ziel.

Sie können auch Blackout-Perioden für Zeiten festlegen, in denen keine Aktivitäten stattfinden sollen. Weitere Informationen dazu finden Sie unter Festlegen einer Blackout-Periode für ein Ziel. Sie können bestimmte Packages auch von der Blackout-Periode ausnehmen. Weitere Informationen dazu finden Sie unter Ausnehmen von Packages von der Blackout-Periode.

Hinweis: Geplante Aktivitäten finden nicht statt, wenn die Tuner-Eigenschaft für die Blackout-Periode (marimba.schedule.filter) auf den Clients auf Nie gesetzt wurde.

Was sind Primär- und Sekundärzeitpläne?

Primär- und Sekundärzeitplan bestimmen, wann die Primär- und der Sekundärstatus eines Package von den Zielen in einer Policy übernommen werden sollen. Die Zeitpläne umfassen Aktivierungs- und Verfalldaten sowie Zeiten, zu denen Primär- und Sekundärstatus des Package von Zielen übernommen werden sollen. Sie müssen einen Sekundärstatus und Sekundärzeitplan nur angeben, wenn als Wert des primären Status Bereitstellen oder Ankündigen angegeben ist. Eine Liste der verschiedenen Primär- und Sekundärstatus finden Sie unter Übersicht über Installationsstatus.

Beispiel: Sie können Primär- und Sekundärzeitpläne so festlegen, dass ein Package zu einer bestimmten Zeit auf die Ziele heruntergeladen und zu einer anderen, späteren Zeit installiert wird. Dazu könnten Sie den Primärstatus eines Package zunächst auf Bereitstellen setzen, so dass es zwischen dem 1. Januar, 17:00 Uhr (Aktivierung) und dem 14. Januar, 17:00 Uhr (Verfallszeit) auf die Ziele heruntergeladen (aber noch nicht installiert) wird. Danach könnten Sie den Sekundärstatus auf Installieren setzen, so dass das Package zwischen dem 14. Januar, 17:00 Uhr (Aktivierung) und dem 28. Januar, 17:00 Uhr (Verfallszeit) installiert wird.

Wenn die Policy viele Packages beinhaltet, können Sie einen Sekundärzeitplan nur festlegen, wenn für ein oder mehr Packages ein Sekundärstatus angegeben ist. Der Sekundärzeitplan wird nur Packages mit Sekundärstatus zugewiesen.

Eine Anleitung zum Festelegen der Primär- und Sekundärzeitpläne finden Sie unter Festlegen des Primär- und Sekundärzeitplans für Packages.

Was sind Aktivierung und Verfall?

Aktivierung und Verfall definieren des Zeitraums, innerhalb dessen Sie Packages für bestimmte Aktivitäten oder Zustände verfügbar machen möchten.

Aktivierung bestimmt den Zeitpunkt, ab dem Installationsstatus, Aktualisierungen oder Reparaturaktivitäten eines Package von den Zielen innerhalb einer Policy übernommen werden können. Beispiel: Wenn Sie ein bestimmtes Package nach dem 31. Januar, 17:00 Uhr auf Ziele verteilen möchten, können Sie dies in der Policy als Aktivierungszeitpunkt angeben. Sie können auch festlegen, dass die Aktivierung stattfindet, sobald Policy Service aktualisiert wird. Wenn Sie diese Optionen auswählen, finden die Aktivitäten unmittelbar bei der nächsten Policy Service-Aktualisierung statt. Weitere Informationen zu Policy Service-Aktualisierungen finden Sie unter Angeben des Policy Service-Zeitplans für ein Ziel.

Verfall bestimmt den Zeitpunkt, ab dem der Installationsstatus, Aktualisierungen oder Reparaturaktivitäten eines Package von den Zielen innerhalb einer Policy nicht mehr übernommen werden können. Das bedeutet, dass Aktivitäten, die nach Ablauf der Verfallszeit noch nicht stattgefunden haben, auch nicht mehr stattfinden werden. Beispiel: Wenn es nach dem 28. Februar, 17:00 Uhr nicht mehr möglich sein soll, ein Package herunterzuladen, können Sie dies in der Policy als Verfallszeit angeben. Clients können das Package nach diesem Zeitpunkt nicht mehr herunterladen. Sie können auch angeben, dass für ein bestimmtes Package keine Verfallszeit bestehen soll.

Hinweis: Nach der Verfallszeit werden Packages aber nicht von den Clients entfernt, auf denen sie bereits installiert sind. Betroffen sind nur Clients, die eine bestimmte Aktivität, z. B. eine Installation, bis zum Verfallszeitpunkt noch nicht durchgeführt haben.

Sie können für Aktivierung und Verfall von Aktivitäten, wie beispielsweise Aktualisierungen und Reparaturaktivitäten, auch einen sich wiederholenden Zeitplan angeben. In diesen Fällen bestimmt "Aktivierung" den Zeitpunkt, nach dem Aktualisierungen und Reparaturaktivitäten regelmäßig stattfinden sollen. "Verfall" bestimmt dagegen den Zeitpunkt, nach dem sie nicht mehr stattfinden sollen.

Was ist ein Aktualisierungszeitplan?

Der Aktualisierungszeitplan bestimmt, wann ein Package aktualisiert wird, und installiert die Aktualisierung, wenn erforderlich, automatisch. Der Aktualisierungszeitplan ist nützlich, wenn die Applikationen und Inhalte, die Sie packen und an Benutzer verteilen, häufig wechseln. Sie können den Aktualisierungszeitplan dazu verwenden, regelmäßig nach neuen oder geänderten Dateien zu suchen und diese herunterzuladen. Der Aktualisierungszeitplan hat lediglich dann Auswirkungen auf den Client, wenn bereits eine Subscription für das Package eingerichtet oder wenn das Package bereits installiert ist.

Hinweis: Das Planen von Package-Aktualisierungen kann Einfluss auf die Software auf den Clients haben und erfordert eine Berücksichtigung der Best Practices. Weitere Informationen dazu finden Sie unter Verwalten von Software, Daten und Aktualisierungen.

Mit Policy Manager können Sie den Aktualisierungszeitplan außer Kraft setzen, der während der Erstellung des Package und der Veröffentlichung festgelegt wurde. Sie können einen sich wiederholenden Aktualisierungszeitplan festlegen, der täglich, wöchentlich oder monatlich an bestimmten Tagen und zu bestimmten Zeiten ausgeführt wird. Sie können für den Aktualisierungszeitplan auch Nie angeben. Optional können Sie Aktivierungs- und Verfallszeitpunkt so festlegen, dass Sie steuern können, wann Aktualisierungen gestartet und gestoppt werden.

Eine Anleitung zum Festlegen des Aktualisierungszeitplans finden Sie unter Festlegen des Aktualisierungszeitplans für Packages.

Hinweis: Für den Aktualisierungszeitplan des Package wird Nie angezeigt, wenn Sie ihn mit Tuner Administrator betrachten. Dies gilt selbst dann, wenn mit Policy Manager ein anderer Zeitplan festgelegt wurde. Policy Service legt für den Aktualisierungszeitplan des Package Nie fest, so dass das Package sich nicht selbst aktualisiert, sondern nach den mit Policy Manager festgelegten Zeitplänen aktualisiert wird.

Was ist ein Reparaturzeitplan?

Der Reparaturzeitplan bestimmt, wann ein Package überprüft und wann es, wenn erforderlich, automatisch repariert wird. Dies ist nützlich, wenn Applikationen und Inhalte, die sie packen und an Benutzer verteilen, unbenutzbar oder nicht korrekt ausgeführt werden. Sie können die Reparaturfunktionen dazu verwenden, automatisch nach Problemen, wie fehlenden oder fehlerhaften Dateien, zu suchen und diese zu beheben. Der Reparaturzeitplan hat lediglich dann Auswirkungen auf den Client, wenn bereits eine Subscription für das Package eingerichtet oder wenn das Package bereits installiert ist. Außerdem funktionieren Reparaturvorgänge nur bei Packages, die mit BMC CM Application Packager erstellt wurden.

Mit Policy Manager können Sie den Reparaturzeitplan außer Kraft setzen, der während der Erstellung des Package und der Veröffentlichung festgelegt wurde. Sie können einen sich wiederholenden Reparaturzeitplan festlegen, der täglich, wöchentlich oder monatlich an bestimmten Tagen und zu bestimmten Zeiten ausgeführt wird. Sie können für den Reparaturzeitplan auch Nie angeben. Optional können Sie Aktivierungs- und Verfallszeitpunkt so festlegen, dass Sie steuern können, wann Reparaturaktivitäten gestartet und gestoppt werden.

Eine Anleitung zum Festlegen des Reparaturzeitplans finden Sie unter Festlegen des Reparaturzeitplans für Packages.

Was geschieht bei einer Reparatur? Wenn Sie ein Package reparieren, werden zunächst die Datei und andere Objekte im Package mit den Dateiinformationen verglichen, die das Package bei seiner Installation hatte. Nach Abschluss der Überprüfung tritt einer der folgenden Fälle ein:

    • Alle Objekte stimmen überein. Es sind keine Reparaturen erforderlich.
    • Nicht alle Objekte stimmen überein. Der Ziel-Tuner installiert die betroffenen Dateien oder Objekte neu, wenn diese im Tuner-Speicher verfügbar sind. Wenn die betroffenen Dateien nicht verfügbar sind, kontaktiert der Ziel-Tuner den Transmitter, um die Dateien oder Objekte herunterzuladen, die für den Abschluss der Reparatur erforderlich sind.

      • Hinweis: Wenn der Transmitter nicht kontaktiert werden kann oder das Package auf dem Transmitter gelöscht oder verändert wurde, so dass die erforderlichen Objekte nicht mehr verfügbar sind, wird das Objekt, das nicht repariert werden kann, in das Logfile eingetragen.

Was ist eine Wiederholung?

Wiederholung bezieht sich auf Aktivitäten, die wiederholt auftreten, üblicherweise gemäß einem normalen Zeitplan. Sie können in Policy Manager sich wiederholende Zeitpläne für Aktualisierungen und Reparaturaktivitäten festlegen. Solche Aktivitäten finden wiederholt gemäß dem Zeitplan zwischen dem Aktivierungs- und Verfallszeitpunkt statt.

Beispiel: Sie können einen Zeitplan für ein Package so einrichten, dass dieses täglich aktualisiert wird, aber vor dem Aktivierungszeitpunkt Aktualisierungsanfragen nicht beantwortet werden. Nach dem Aktivierungszeitpunkt und vor der Verfallszeitpunkt beantwortet Policy Manager die Aktualisierungsanfragen. Nach dem Verfallszeitpunkt wird der Aktualisierungszeitplan der Applikation auf „Nie“ gesetzt, und weitere Aktualisierungsanfragen werden nicht beantwortet.
