Lorsque vous créez ou modifiez une policy à l'aide du Policy Manager, vous avez la possibilité d'indiquer les propriétés de tuner et de package d'une cible. Les propriétés de tuner et de package vous permettent de contrôler le comportement de la cible ou des packages sur les machines cibles, comme par exemple le nombre de nouvelles tentatives en cas d'échec de téléchargement ainsi que leur fréquence ou bien le comportement de redémarrage d'un package particulier.

Cette section comprend les rubriques suivantes :

    • Définition d'une propriété de tuner
    • Quelles propriétés de tuner paramétrez-vous habituellement avec Policy Manager ?
    • Définition d'une propriété de package
    • Propriétés qui régissent le redémarrage des packages (Windows uniquement)

Définition d'une propriété de tuner

Les propriétés de tuner configurent les nombreuses caractéristiques et paramètres d'un tuner, y compris l'adresse URL de ses channels primaires et l'adresse URL des mises à jour du tuner, ainsi que les détails relatifs aux licences et au proxy. Vous pouvez utiliser Policy Manager pour ajouter, modifier et supprimer les propriétés de tuner du tuner cible (dans le fichier prefs.txt).

Pour les listes des propriétés de tuner que vous pouvez généralement définir à l'aide du Policy Manager, reportez-vous à la rubrique Quelles propriétés de tuner paramétrez-vous habituellement avec Policy Manager ?. Pour obtenir la liste des propriétés de tuner, reportez-vous au chapitre sur les propriétés de tuner dans le Guide de référence de BMC Configuration Management, disponible sur le site Web de l'assistance à la clientèle BMC.

Pour obtenir des instructions concernant le paramétrage des propriétés de tuner, reportez-vous à la section Paramétrage des propriétés de tuner.

Quelles propriétés de tuner paramétrez-vous habituellement avec Policy Manager ?

Si Policy Service est installé, certaines propriétés supplémentaires de tuner sont disponibles. À l'aide de l'une des méthodes suivantes, vous pouvez paramétrer les propriétés de tuner suivantes pour modifier l'interaction du tuner avec Policy Service :

    • Si un programme d'installation de tuner est créé durant la configuration et le déploiement
    • À l'aide d'une mise à jour du profil
    • À l'aide du Tuner Administrator
    • À l'aide du Policy Manager.

Les neuf premières propriétés apparaissant dans le Tableau 4-7 sont celles qui s'affichent dans la liste de la page Propriétés de tuner et de package de Policy Manager. Elles sont indiquées dans le même ordre que celui de leur affichage sur la page. Les propriétés du tableau restantes doivent être entrées avec leur nom de propriété complet. Elles apparaissent par ordre alphabétique.

silent ou aspackaged
 
Valeur par défaut : silent

marimba.subscription.adminusers
<utilisateur1>,
<utilisateur2>,...
désigne une liste d'administrateurs séparés par des virgules. Cette liste d'administrateurs peut contenir le nom de tout utilisateur autorisé à se connecter aux machines cibles.
 
Valeur par défaut : aucune
Si la machine cible de l'utilisateur est équipée du Policy Service 5.0.1 ou une version supérieure, vous pouvez paramétrer cette propriété du tuner afin de spécifier les administrateurs autorisés à se connecter temporairement aux machines d'un utilisateur lorsqu'ils utilisent un ciblage par utilisateur (pour des raisons de dépannage notamment). Cette propriété empêche la suppression des channels de l'utilisateur lorsque vous vous connectez en tant qu'administrateur.
Lorsque l'administrateur se connecte :
  • Les channels d'un autre utilisateur ne sont pas supprimés.
  • Si certains channels ont été attribués à un administrateur, ils sont alors fournis à la machine cible.
  • Les propriétés définies pour l'administrateur sont configurées. Les propriétés définies pour les utilisateurs sont parfois remplacées par celles définies pour l'administrateur. Toutefois, lorsque les utilisateurs se reconnectent et que Policy Service est mis à jour, ils récupèrent leurs propriétés.
marimba.subscription.autostart
<nom_
channel>, “<argument1>, <argument2>, <argument3>“,true ou false
 
Valeur par défaut : aucune valeur
Vous permet de démarrer le channel spécifié une fois Policy Service exécuté.
  • <nom_channel> définit le nom du channel que vous voulez démarrer. Il est obligatoire.
  • « <argument1>,<argument2>,<argument3> »
    sont les arguments que vous voulez transmettre au channel. Ils sont facultatifs.
  • true ou false indique s’il faut mettre à jour le channel avant de le démarrer. Ce dernier argument est facultatif.
 
Exemple :
marimba.subscription.autostart=ChannelManager,”arg1,arg2,arg3”,true
 
Remarque : soyez vigilant lors de l'utilisation de cette propriété avec les channels BMC CM. Certains channels peuvent être configurés pour démarrer Policy Service et provoquer une récurrence.
marimba.subscription.machinename
chaîne représentant le nom de la machine
 
Valeur par défaut : aucune valeur
Vous permet de remplacer le nom de machine renvoyé par Policy Service. Le nom entré ici doit correspondre au nom utilisé dans le fichier plat de la machine. Par défaut, cette propriété ne porte aucune valeur et le nom de machine est simplement le hostname DNS, sans informations de domaine.
marimba.subscription.retryintervalsec
nombre entier
 
Valeur par défaut : 30
(30 secondes)
Cette propriété définit l'intervalle d'attente, en secondes, avant une nouvelle tentative de connexion au LDAP. Cette propriété de tuner est à définir sur le tuner qui héberge le transmitter sur lequel vous avez publié le plugin Policy Service.
marimba.subscription.timeout
nombre entier
 
Valeur par défaut : 3 600
(3 600 secondes ou 1 heure)
Cette propriété définit la période maximale (exprimée en secondes) pendant laquelle Policy Service doit attendre la fin d'une opération (comme par exemple l'envoi d'une notification de date par un package) avant de continuer.
marimba.subscription.varytime
nombre entier
 
Valeur par défaut : 10
(10 minutes)
Cette propriété permet de définir la durée maximale en minutes pendant laquelle des événements planifiés (comme par exemple le téléchargement ou la mise à jour d'un package) peuvent être reportés. Cela permet d'améliorer les performances des transmitters via le dispersement des requêtes des machines cibles pendant les périodes de trafic dense. Par exemple, si cette propriété porte la valeur 10 minutes et qu'un événement est planifié à 10h00 du matin, l'événement peut avoir lieu à tout moment entre 10h00 et 10h10.
marimba.tuner.display.noerrors
true ou false
 
Valeur par défaut : false
Cette propriété indique si le tuner doit afficher des messages d'erreur et d'avertissement. Pour masquer l'affichage des messages d'erreur et d'avertissement, affectez à cette propriété la valeur true. Les messages d'erreur et d'avertissement sont alors imprimés sur une console système. Notez que cette propriété ne s'applique pas si vous utilisez le tuner sans affichage.
marimba.tuner.display.noprogress
true ou false
 
Valeur par défaut : false
Cette propriété indique si le tuner affiche une barre de progression lors d'un abonnement ou d'une mise à jour des channels. Pour masquer les barres de progression, affectez à cette propriété la valeur true. Notez que cette propriété ne s'applique pas si vous utilisez le tuner sans affichage.

Définition d'une propriété de package

Les propriétés de package fournissent des informations sur le package et la façon dont il doit s'exécuter. Elles sont aussi appelées propriétés de channels. Ces informations sont généralement employées par le tuner cible. Les packages, habituellement des applications mises en package dans des channels (à l'aide de Application Packager), ont des propriétés supplémentaires qui sont appelées les paramètres de channel. Les Paramètres de channel fournissent les informations et les paramètres utilisés lors de l'installation et de l'exécution des packages.

Les propriétés et les paramètres de package sont stockés respectivement dans les fichiers properties.txt et parameters.txt. Ces fichiers sont généralement créés (dans le répertoire du channel) en même temps que le channel ou le package. Vous pouvez modifier ces fichiers manuellement ou utiliser Channel Copier ou Publisher.

Pour obtenir la liste des propriétés de package et des paramètres de channel, reportez-vous aux chapitres sur les propriétés et paramètres de channel dans le Guide de référence de BMC Configuration Management, disponible sur le site Web de l'assistance à la clientèle BMC.

Les propriétés et paramètres de package s'affichent généralement dans le format suivant :

<nom_propriété>=<valeur_propriété>

Pour obtenir des instructions concernant le paramétrage des propriétés de package, reportez-vous à la section Paramétrage des propriétés de package.

Propriétés qui régissent le redémarrage des packages (Windows uniquement)

Lors de l'installation des packages créés à l'aide de l'Application Packager, certains fichiers ne peuvent pas être inscrits du fait qu'ils ont été verrouillés par le système d'exploitation. La plupart des incidents de fichiers verrouillés sont dus à un fichier ouvert ou à un DLL actuellement chargé. Dans de tels cas, le nouveau fichier est enregistré sur le disque sous un nom temporaire et désigné comme devant être renommé avec le nom et l'emplacement correspondants lors du redémarrage du système.

Policy Service peut installer plusieurs packages lors d'une seule mise à jour de policy. Si plusieurs packages ont été installés en une session, Policy Service reporte le redémarrage du système jusqu'à la mise à jour et l'installation complètes de tous les packages. En fonction de l'état d'avancement de l'installation, le redémarrage peut s'effectuer même en cas d'échec de l'installation.

Du côté du Policy Service, le comportement de redémarrage est contrôlé par les propriétés de tuner marimba.subscription.reboot et marimba.subscription.reboot.time. Le Policy Service ne vérifie le redémarrage que si la propriété marimba.subscription.reboot est définie sur true, quels que soient les paramètres de chaque package.

Policy Service peut prendre en compte les propriétés de redémarrage de chaque package créé par Application Packager. Ces propriétés de packages sont décrites dans le Tableau 4-8. Deux de ces propriétés, reboot.allow et reboot.force définissent le comportement de redémarrage, comme présenté dans le Tableau 4-9. Les deux autres propriétés de package, reboot.showdialog et reboot.allowcancel, permettent de contrôler la boîte de dialogue qui est affichée lorsqu'un redémarrage est requis.

Tableau 4-8:  Propriétés de redémarrage de package 
Nom de propriété
Valeurs possibles
Description
reboot.allow
true ou false
 
Valeur par défaut : true
Ce paramètre indique que le channel peut redémarrer la machine si nécessaire. Par défaut, il est défini sur true. Définissez ce paramètre sur false si ce comportement n’est pas requis.
Remarque : les channels ne forcent pas les machines à redémarrer automatiquement en mode silencieux, sauf si le paramètre reboot.force est défini sur true.
reboot.allowcancel
true ou false
 
Valeur par défaut : true
Si vous définissez ce paramètre sur true et qu’un redémarrage est requis au niveau de la machine cible, une option permettant à l’utilisateur d’annuler le redémarrage apparaît dans la boîte de dialogue de redémarrage (en mode IU complet ou semi-silencieux). Si le programme d’installation est en mode IU silencieux, alors aucun redémarrage n'est effectué, même si un redémarrage est nécessaire.
reboot.force
true ou false
 
Valeur par défaut : false
Si vous le définissez sur true, ce paramètre force la machine à redémarrer à la fin de l'installation. Par défaut, il est défini sur false. Ce paramètre doit être utilisé avec précaution, car il risque d’entraîner des interactions indésirables avec l’utilisateur.
reboot.showdialog
Valeurs valides : true ou false
 
Valeur par défaut : true
 
Ce paramètre indique qu’en mode interactif (mode autre que silencieux ou semi-silencieux), le channel doit afficher la boîte de dialogue de redémarrage, au besoin. Par défaut, il est défini sur true. Définissez ce paramètre sur false pour modifier ce comportement.

Si les propriétés reboot.allow et reboot.force sont toutes les deux définies sur true pour tous les packages installés par Policy Service, ce dernier redémarre le système que cela soit requis ou non.
